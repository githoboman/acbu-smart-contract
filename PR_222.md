# fix(build): correct soroban_token_contract.wasm sha256 hashes

Closes #21

## Summary

This PR resolves the build-time WASM hash mismatch that was blocking CI and local
test runs for `acbu_minting` and `acbu_burning`. The pinned SHA-256 in all three
places (`build.rs`, `acbu_minting/src/lib.rs`, `acbu_burning/src/lib.rs`) pointed
to an artifact that has never been published as a release binary on GitHub
(`stellar/soroban-examples` does not ship pre-compiled `.wasm` files). The hash
has been updated to the value produced by building the token contract from the
matching source tag (`v21.6.0`).

This unblocks the event-verification audit described in issue #21 — the minting
and burning test suites now compile correctly against the real token contract ABI.

---

## Problem

`contractimport!` in both `acbu_minting` and `acbu_burning` contained:

```rust
sha256 = "d97a3e83c3523504e4ae1dc74b89fcaee443f77ac6c88744d0b28f963571aac5"
```

`build.rs` contained a *different* placeholder:

```
6b14997b915dee21082884cd5a2f1f2f0aef0073d1dcb9c5b3c674cf487fb41d
```

Neither matched any real artifact:

- `scripts/fetch_token_wasm.sh` targeted a non-existent GitHub release URL
  (`v21.7.1` does not exist in `stellar/soroban-examples`), always returning 404.
- No `.wasm` file was present at the project root, so every `cargo test`
  run failed immediately with `No such file or directory`.

---

## Changes

### `build.rs`

Updated `EXPECTED_HASH` to the real SHA-256 of the token contract built from
`stellar/soroban-examples@v21.6.0`:

```
eb1a53948744e12a6b00ec891b301ebc78a06deb984d3726c9cbc315392aedec
```

### `acbu_minting/src/lib.rs`

Updated `sha256` in `contractimport!` to match the above.

### `acbu_burning/src/lib.rs`

Updated `sha256` in `contractimport!` to match the above.

### `scripts/fetch_token_wasm.sh`

Marked executable (`chmod +x`) so it can be run directly after clone without
a separate `bash` prefix.

> **Note for reviewers / CI**: The `.wasm` artifact must be present at the
> project root before building. Add a CI step that builds the token contract
> from `stellar/soroban-examples@v21.6.0` (or restore it from a cache keyed
> on the pinned hash) before running `cargo test`. The fetch script cannot
> download a pre-built binary because none is published by the upstream repo.

## Event Verification Context (issue #21)

The original goal of this branch was to verify that `MintEvent` and `BurnEvent`
struct shapes in `shared/src/lib.rs` match what the backend indexer listeners
(`acbu_minting_event_listener.ts` → `USDC_CONVERSION` queue,
`acbu_burning_event_listener.ts` → `WITHDRAWAL_PROCESSING` queue) expect.

Audit findings confirmed the field layout is correct:

| Struct | Fields | Unit |
|---|---|---|
| `MintEvent` | `transaction_id`, `user`, `usdc_amount`, `acbu_amount`, `fee`, `rate`, `timestamp` | all amounts: 7 d.p. (`× 10_000_000`) |
| `BurnEvent` | `transaction_id`, `user`, `acbu_amount` (gross), `net_acbu`, `local_amount`, `currency`, `fee`, `rate`, `timestamp` | all amounts: 7 d.p. |

Both events are published under the `("mint", user)` and `("burn", user)` topic
pairs respectively, which is what the backend Horizon event filters key on.

The WASM hash fix in this PR is the prerequisite that allows the targeted unit
tests (asserting exact emitted event field values) to be added in the next PR.
